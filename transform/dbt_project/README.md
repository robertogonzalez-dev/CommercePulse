# CommercePulse — dbt Project (Phase 3)

End-to-end dimensional warehouse modelled with dbt on DuckDB.
Transforms Bronze raw tables into production-quality Silver staging models and Gold analytical marts.

---

## Architecture Overview

```
Bronze (DuckDB)          Silver (dbt views)          Gold (dbt tables)
─────────────────        ──────────────────          ─────────────────────────────
raw_customers      ───►  stg_customers         ──►  dim_customer
raw_products       ───►  stg_products          ──►  dim_product

raw_order_items    ───►  stg_order_items       ──►  dim_channel (conformed)
```

## Star Schema Design

### Fact Table Grains

| Fact Table               | Grain                              | Key Business Questions                           |
|--------------------------|------------------------------------|-------------------------------------------------|
| `fact_orders`            | One row per order                  | Revenue, AOV, order volume, refund rates         |
| `fact_order_items`       | One row per order line item        | Product sales, margin, discount analysis         |
| `fact_payments`          | One row per payment transaction    | Payment method mix, success rates, reconciliation|
| `fact_refunds`           | One row per refund                 | Refund reasons, financial impact                 |
| `fact_inventory_snapshots`| One row per product/warehouse/date | Stock levels, inventory value, reorder risk      |
| `fact_sessions`          | One row per web session            | Conversion funnel, channel traffic, device mix   |
| `fact_marketing_spend`   | One row per campaign               | ROAS, CPA, channel efficiency                    |

### Dimension Keys

All surrogate keys are MD5 hashes generated by `dbt_utils.generate_surrogate_key`.
`dim_date` uses an integer `date_key` in `YYYYMMDD` format.

## Materialization Strategy

| Layer        | Materialization | Rationale                                                              |
|--------------|-----------------|------------------------------------------------------------------------|
| Staging      | `view`          | Bronze tables are small; views cost nothing and always reflect latest   |
| Intermediate | `view`          | Logic holders only; downstream tables pay the compute cost once         |
| Dims (Gold)  | `table`         | Stable, small, joined frequently — should be pre-computed               |
| Facts (Gold) | `table`         | Aggregations at query time on views would be too slow for analysis      |
| Reporting    | `table`         | Pre-aggregated for dashboard tools; rebuilt on each full run            |

> **Note:** For production workloads with millions of rows, `fact_orders`,
> `fact_order_items`, and `fact_sessions` would be upgraded to `incremental`
> with an `updated_at` or `ordered_at` watermark. For this portfolio project
> with small seed data, `table` is correct and simpler.

## Setup

### Prerequisites

```bash
pip install dbt-duckdb dbt-core
```

### Install dbt packages

```bash
cd transform/dbt_project
dbt deps
```

### Configure connection

**Option A** — Copy `profiles.yml` to `~/.dbt/profiles.yml`

**Option B** — Run with profiles dir flag (recommended for portfolio):
```bash
dbt run --profiles-dir .
```

---

## Running the Project

```bash
# From transform/dbt_project/

# Full build: all models in correct DAG order
dbt run --profiles-dir .

# Build a specific layer
dbt run --profiles-dir . --select staging
dbt run --profiles-dir . --select marts.core
dbt run --profiles-dir . --select marts.reporting

# Run all tests
dbt test --profiles-dir .

# Test a specific model
dbt test --profiles-dir . --select fact_orders

# Generate and serve documentation
dbt docs generate --profiles-dir .
dbt docs serve --profiles-dir .

# See the lineage DAG
dbt ls --profiles-dir . --select +mart_sales_summary

# Full clean rebuild
dbt clean && dbt deps && dbt run --profiles-dir . && dbt test --profiles-dir .
```

## Key Business Metrics Defined

| Metric                    | Definition                                                     | Location                      |
|---------------------------|----------------------------------------------------------------|-------------------------------|
| Gross Revenue             | `net_subtotal + shipping_cost`                                 | `int_order_financials`        |
| Net Revenue               | `gross_revenue - amount_refunded`                              | `int_order_financials`        |
| Average Order Value (AOV) | `gross_revenue / total_fulfilled_orders`                       | `mart_sales_summary`          |
| Refund Rate               | `orders_with_refunds / fulfilled_orders * 100`                 | `mart_sales_summary`          |
| Cancellation Rate         | `cancelled_orders / total_orders * 100`                        | `mart_sales_summary`          |
| Discount Rate             | `total_discounts / gross_subtotal * 100`                       | `mart_sales_summary`          |
| Historical CLV            | `sum(net_revenue)` per customer                                | `mart_customer_ltv`           |
| Predicted CLV (2yr)       | `avg_order_value × annual_frequency × 2`                       | `mart_customer_ltv`           |
| ROAS                      | `revenue_attributed / amount_spent`                            | `fact_marketing_spend`        |
| CPA                       | `amount_spent / conversions`                                   | `fact_marketing_spend`        |
| Session Conversion Rate   | `converted_sessions / total_sessions * 100`                    | `mart_channel_performance`    |
| Bounce Rate               | `bounced_sessions / total_sessions * 100`                      | `mart_channel_performance`    |
| Contribution Margin       | `net_line_total - (quantity × cost_price)`                     | `fact_order_items`            |
| Realized Margin %         | `total_contribution_margin / net_revenue * 100`                | `mart_product_performance`    |
| Days of Cover             | `quantity_available / avg_daily_units`                         | `mart_inventory_risk`         |
| RFM Score                 | Recency + Frequency + Monetary (1–5 each, `ntile(5)`)          | `mart_customer_ltv`           |

## Project Structure

```
transform/dbt_project/
├── dbt_project.yml          # Project config and materialization defaults
├── packages.yml             # dbt_utils dependency
├── profiles.yml             # DuckDB connection (copy to ~/.dbt/ or use --profiles-dir .)
├── models/
│   ├── staging/             # Silver: 1:1 with Bronze; clean, cast, deduplicate
│   ├── intermediate/        # Silver: pre-join logic shared by multiple Gold models
│   └── marts/
│       ├── core/            # Gold: dims + facts (star schema)
│       └── reporting/       # Gold: pre-aggregated business marts
├── macros/
│   ├── generate_schema_name.sql   # Prevents dbt schema prefixing (e.g. main_gold → gold)
│   └── safe_divide.sql            # Null-safe division helper
└── tests/
    ├── generic/             # Reusable test macros
    └── singular/            # One-off data quality assertions
```

## Documenting Lineage in GitHub

1. Run `dbt docs generate --profiles-dir .` — this produces `target/manifest.json` and `target/catalog.json`
2. Commit the `target/` directory (add an exception in `.gitignore` for `target/manifest.json`)
3. Host the docs site via GitHub Pages or Netlify by serving the `target/` directory
4. Add a badge to your root `README.md`:

```markdown
[![dbt docs](https://img.shields.io/badge/dbt-docs-orange)](https://your-username.github.io/CommercePulse/)
```

Alternatively, embed a lineage screenshot by running `dbt docs serve`, navigating to the DAG view, and saving a screenshot to `docs/lineage.png`.
